Why PrismaAdapter?
Connects NextAuth to Postgres so every Google login creates/updates a User row automatically. Without it, sessions exist only in JWT — no user row in DB, no way to associate problems with a user.

Why session callback?
NextAuth v5 default session only includes name, email, image. The callback explicitly attaches user.id so all API routes can identify who is making the request.

Why PrismaPg adapter in lib/prisma.ts?
Prisma 7 moved to a driver adapter model. Plain new PrismaClient() throws PrismaClientInitializationError. PrismaPg is the official PostgreSQL adapter — required for Prisma 7 with Supabase.

Why globalThis singleton?
Next.js hot reloads create new module instances on every save. Without storing the Prisma client on globalThis, each reload opens a new DB connection. Postgres has a connection limit — this prevents exhausting it in development.

Why AUTO vs CUSTOM folder types?
Sync should only touch AUTO folders — user-created CUSTOM folders (like "Google Prep") must never be overwritten by a sync. The type enum enforces this at the data model level, not just in application logic.

Tag source: LeetCode's question GraphQL query returns official topic tags per problem. We fetch tags per titleSlug during sync rather than using AI classification — deterministic, free, and uses the same taxonomy LeetCode users already know.

Primary tag selection for folder assignment uses LeetCode's own skill tier system as priority: Advanced tags (DP, Backtracking, Divide and Conquer) beat Intermediate tags (Sliding Window, Hash Table, Math) which beat Fundamental tags (Array, String, Sorting). This mirrors LeetCode's own categorization so users see folders organized by the same mental model they already know from the platform.

LeetCode sync pipeline: fetch recent submissions → for each problem fetch topic tags via GraphQL → run getPrimaryTag() with tier priority → upsert AUTO folder → upsert Problem → link via FolderProblem join table. Two separate GraphQL calls per problem because recentAcSubmissionList doesn't return tags — this is a known LeetCode API limitation. At 20 problems this is 40 API calls total, acceptable for now. At scale this would need a queue with rate limiting

Why Platform/Difficulty as real Prisma enums instead of plain strings?
A free-text `platform: String` field lets application code write any string — a typo like "Leetcode" (wrong case) silently creates a second, inconsistent value that filters/indexes miss. Making them Postgres enums pushes that validation down to the database: an invalid value is now rejected at the INSERT/UPDATE, not discovered later as a data quality bug. Tradeoff: adding a new platform (e.g. "HackerRank") now requires a migration instead of just writing a new string, which is the right tradeoff for a fixed, small set of values.

Why bounded concurrency (8 in flight) instead of sequential or fully parallel for LeetCode tag fetching?
The sync fetches topic tags with one GraphQL call per problem (see pipeline above) — originally awaited one at a time in a for-loop, so 20 problems meant 20 sequential round trips just for tags. Firing all of them at once with Promise.all is the opposite failure mode: LeetCode's API isn't ours to control, and 40+ simultaneous requests risks 429s or an IP flag. A small worker pool (mapWithConcurrency, limit 8) caps how many requests are in flight at once — most of the speedup of full parallelism, without hammering an external API we don't own.

Why does /api/sync/leetcode respond immediately and do the real work in next/server's after()?
Vercel serverless functions have a request time limit; a sync that awaits 20+ GraphQL calls plus DB writes inside the request risks timing out, especially as a user's problem count grows. after() runs a callback after the HTTP response has already been sent, so the client gets an instant "sync started" response while the actual fetching/upserting continues server-side. codeforces/route.ts already used this pattern — leetcode/route.ts didn't, which was an inconsistency as much as a performance problem.

Why updateMany + userId in the WHERE clause instead of update by id alone (notes/save, and the ownership checks in revision/toggle, problem/move)?
prisma.problem.update({ where: { id } }) only accepts a unique key — there's no way to also require userId in that same where clause. That means any authenticated user who knows (or guesses/enumerates) a problemId can modify a row they don't own — a textbook IDOR (Insecure Direct Object Reference) bug. updateMany's WHERE clause has no such restriction, so { id, userId } together scope the write correctly; a 0-row result means "not found or not yours," and both cases should 404 rather than leak which one it was.

Why does folder/delete take an explicit action ("delete_problems" | "move_problems") instead of just deleting?
A folder's problems don't only exist in that folder in general — an AUTO topic folder and a CUSTOM "Revision" folder can both point at the same Problem row via FolderProblem. Silently cascade-deleting every problem in a folder would also delete problems that are still organized elsewhere. The two explicit modes make the choice visible to the user at delete time: move problems to another folder first, or only delete the ones that don't exist anywhere else (computed via a groupBy count on FolderProblem per problemId).

Migration drift recovery (2026-07-12): a migration that converted platform/difficulty from text to enums failed partway (P3018) because its DROP INDEX statement assumed an index name ("Problem_url_key") that didn't exist on the live database. Root cause: the live schema had already diverged from the migration history — a composite unique index (userId, url) existed on the DB without a corresponding migration file, meaning someone previously ran a schema change (most likely `prisma db push`) that updates the database directly without writing a migration. Diagnosed by querying pg_indexes and pg_type directly rather than assuming the migration folder was accurate. Lesson: migration files are only a reliable source of truth if every schema change goes through them — `db push` is fine for prototyping but breaks that guarantee for any environment with real data.

Why JWT sessions instead of database sessions (2026-07-12)?
NextAuth needs to answer "who is this request from" on every page load. With a PrismaAdapter and no explicit `session.strategy`, NextAuth defaults to database sessions: the cookie holds an opaque token, and every request queries the Session table to validate it and look up the user — one DB round-trip before the page's own data queries even start. Switching to `session: { strategy: "jwt" }` moves that check to signature verification instead: the cookie holds a signed token containing the user id directly, verified in-process with no DB call. The adapter is still used for OAuth account linking (Google sign-in still creates/links the User row correctly) — only session *validation* changed. Tradeoff: a database session can be revoked instantly (delete the row, user is logged out everywhere on their next request); a JWT session can't be killed early since there's no row to delete — it stays valid until it expires. Accepted for this app since it's a personal tracker with no payment/account-takeover-sensitive actions where instant revocation matters.

Why replace window.location.reload()/.href with router.refresh()/router.push() (2026-07-12)?
AddProblemModal, ProfileForm, and ProblemList's move-to-folder all forced a full hard browser navigation after a successful action — re-downloading and re-parsing the entire JS bundle and remounting the whole app, instead of Next's client-side router re-fetching just the current route's data and patching the DOM. Purely additive latency with no upside; there was never a reason these needed a hard reload. router.refresh() re-runs the current route's server-side data fetch in place; router.push() is used where the action also navigates to a different route (sync → dashboard).

Why does "Needs Revision" recompute on every dashboard load instead of a scheduled job (2026-07-13)?
Unlike the topic AUTO folders, which only change when a sync runs (a real user action), "needs revision" changes purely from time passing — a problem can go stale with nobody doing anything. The textbook-correct version of this is a scheduled job (cron / worker) that periodically recomputes it in the background, independent of anyone visiting the app. This app has no job-scheduling infrastructure, and standing one up just for this felt like solving a problem this project doesn't have yet. Instead, syncNeedsRevisionFolder() runs synchronously on every dashboard load: it's self-healing (always reflects current staleness, no risk of a stale cron result) at the cost of a write happening inside what's otherwise a page read — a deliberate simplification, not an oversight. If this app ever needs true background recomputation (e.g. to power an email/push notification independent of the user opening the dashboard), that's the point where a real scheduled job becomes worth the complexity.

Why does staleness fall back through lastViewedAt → solvedAt → createdAt instead of just checking lastViewedAt?
ProblemProgress rows (and therefore lastViewedAt) are only created the first time a user clicks through to a problem after this feature shipped — a problem solved before that, and never revisited since, would have no ProblemProgress row at all and would be silently invisible to the staleness check forever. Falling back to solvedAt (and createdAt as a last resort for problems added without a solvedAt) means every problem is eligible from day one, not just ones touched after this feature existed.

Why add CI now instead of earlier (2026-07-13)?
A GitHub Actions workflow (.github/workflows/ci.yml) now runs install → prisma generate → lint → build on every push/PR to master. This is placed after the schema/security/performance fixes deliberately: CI's whole value is catching a broken build before it reaches master, and the original broken-schema commit this session started from is exactly the kind of thing it would have caught. Adding it now that the app actually builds cleanly means the first CI run is a real green baseline, not something immediately red that has to be worked around. The workflow uses placeholder env vars rather than real secrets — prisma generate only needs the schema file, and next build doesn't prerender the auth-gated dynamic routes, so no live DB or API connection is required just to verify the code compiles.

Why is DailyGoal a separate model from the existing Goal model (2026-07-13)?
Goal (weeklyEasy/weeklyMed/weeklyHard + doneEasy/doneMed/doneHard) is a different concept — numeric difficulty-based targets — and was never wired up to any route either. DailyGoal is a free-form checklist item (title + done + a target day), which needed its own shape rather than repurposing a model built for a different kind of tracking. Both now exist unused-or-partially-used in the schema; Goal remains a known gap, not something this feature quietly absorbed.

Why does the buffer list recompute on read instead of a cron job, and why a 2-week cap (2026-07-13)?
Same reasoning as "Needs Revision": no scheduled-job infrastructure exists in this app, so getBufferGoals() just queries "not done, targetDate in the past, within the last 14 days" every time it's called rather than something maintaining a persistent bucket in the background. Goals never get deleted by a cleanup job — once a goal falls outside the 14-day window it simply stops appearing in this query's results, it's still a row in the database. The cap itself is a product decision (made with the user, not assumed): unbounded rollover risked an ever-growing, demoralizing backlog; a 2-week window keeps the buffer meaningful without needing an explicit "give up on this goal" action from the user.

Why is the buffer-day reminder a plain boolean check instead of a real notification system (2026-07-13)?
"Notify the user on their chosen day" could mean an email/push notification reaching them even when the app is closed (needs a real notification channel plus a scheduled trigger — genuine new infrastructure), or a popup that shows only if they happen to open the app that day (zero new infrastructure). Chose the latter, explicitly, with the user: shouldShowBufferReminder() is just `today's day-of-week === user.bufferDay AND buffer list is non-empty` — a boolean the /goals page checks on every load and BufferReminderPopup renders from. If this app ever needs a reminder that reaches someone without them opening it, that's a materially bigger feature (email service + Vercel Cron), not a small addition to this one.

Why does "Needs Revision" now read nextReviewAt instead of only the flat 30-day heuristic (2026-07-13)?
Added real SM-2 (SuperMemo-2, the Anki algorithm) via lib/spacedRepetition.ts and POST /api/problem/review: a user rates recall as Again/Hard/Good/Easy (mapped to the 0-5 scale SM-2 expects), and the response — easeFactor, intervalDays, nextReviewAt — is persisted on ProblemProgress. This gives each problem its own real schedule instead of every problem going stale on the same flat 30-day timer regardless of how well the user actually knows it. getStaleProblems() in revisionEngine.ts now checks nextReviewAt first when it's set (i.e. the user has rated the problem at least once), and only falls back to the original lastViewedAt/solvedAt/createdAt + 30-day heuristic when nextReviewAt is null — same fallback-chain pattern as before, so problems solved before this feature existed, and never rated since, still surface instead of being silently exempt forever.

Why extend ProblemProgress instead of a new table for spaced-repetition state (2026-07-13)?
easeFactor, intervalDays, and nextReviewAt are per-problem review state — exactly what ProblemProgress already models (lastViewedAt, revisionCount, mistakeCount, retentionScore were scaffolded here originally but never wired up to anything). Reusing revisionCount as SM-2's repetition counter and lastRevisedAt as the review timestamp meant three new columns instead of a whole new table plus a join back to Problem.

Why Again/Hard/Good/Easy buttons instead of asking for a raw 0-5 quality score (2026-07-13)?
SM-2 internally wants a 0-5 number, but nobody intuitively knows what "3" means when rating their own recall. Anki's four-button labeling is a well-tested UX solved problem — mapped here to quality 1/3/4/5 respectively (QUALITY constant in lib/spacedRepetition.ts) rather than reinventing a rating scale.

Why native HTML5 drag-and-drop for folder reordering instead of a library (2026-07-13)?
No drag-and-drop library was in package.json, and the one obvious historically-popular option (react-beautiful-dnd) is unmaintained and has known issues with React 18+/19. The dashboard only needed one behavior — drag a tile, see the grid reflow live, persist the final order — which the native draggable/onDragStart/onDragOver/onDragEnd events handle directly without adding a dependency. Reorder is optimistic (local state reflows immediately in FolderGrid.tsx) and persisted via POST /api/folder/reorder, which assigns `order` as array index after an ownership check on every folder id in the request.

Why can't "Needs Revision" and "Revision" be dragged (2026-07-13)?
Both are system-managed — "Needs Revision" is rebuilt by syncNeedsRevisionFolder() on every dashboard load, "Revision" is built by the ★ toggle — and both already float to the top of the dashboard via the existing PINNED_FOLDER_NAMES sort in app/dashboard/page.tsx, independent of the `order` column. Making them draggable would let a user "reorder" a folder whose position is actually hardcoded elsewhere, which would just look broken (the pin sort would silently undo the drag on next load). FolderGrid.tsx excludes both by name from `draggable` and from the id list sent to /api/folder/reorder.

Why cache syncNeedsRevisionFolder in Redis instead of just accepting the write-on-every-load cost (2026-07-13)?
This was flagged as the single heaviest thing the app does on a normal page load: every /dashboard visit ran a full read of the user's problems plus a folderProblem createMany+deleteMany, regardless of whether anything had actually changed since the last visit. A scheduled job would be the textbook fix, but this app still has no job infrastructure (see the earlier "Needs Revision" entry above), and standing one up just for this is the same premature-complexity tradeoff already rejected once. Caching the {count, folderId} result in Redis for 1 hour (lib/revisionEngine.ts) gets most of the benefit for near-zero new complexity: a cache hit skips Prisma entirely. Tradeoff: staleness can now lag up to an hour (e.g. rating a problem via SM-2 won't drop it out of "Needs Revision" until the cache expires or the hour passes) — accepted as the standard cache/consistency tradeoff, same shape as the AI hint cache in app/api/hint/rag/route.ts.

Why a Redis SET NX cooldown instead of the GET-then-INCR pattern already used for AI hints (2026-07-13)?
Added per-user cooldowns on both sync routes (5 minutes) so repeated "Sync" clicks can't hammer LeetCode/Codeforces's APIs or trigger redundant DB upsert passes back-to-back. The AI hint route's rate limit is a *counter* (needs to tell the user "3 of 10 used today"), which is why it reads the count before incrementing. A cooldown is a plain binary gate — "allowed" or "not yet" — with no count to report, so lib/rateLimit.ts's checkCooldown() uses a single `SET key NX EX ttl` instead: the SET only succeeds if the key doesn't already exist, making the check-and-set one atomic operation instead of two, which closes a race window a separate GET-then-SET would have (two requests arriving within milliseconds of each other could otherwise both read "not on cooldown" before either write lands).

Why cursor pagination instead of offset (skip/take) for a folder's problem list (2026-07-13)?
A topic folder like "Array" can realistically grow into the hundreds of problems over years of solving — the folder page previously fetched every problem in a folder in one unbounded query. Offset pagination (`skip: N`) makes Postgres walk and discard the first N rows on every request, which gets slower the further a user pages. A cursor (`cursor: { id }, skip: 1` in Prisma, translating to "rows after this specific row" in the given order) costs about the same regardless of how deep you are, since it's answered directly off the already-indexed `folderId` column rather than by counting through everything before it. app/api/folder/[folderId]/problems/route.ts serves pages of `FOLDER_PROBLEMS_PAGE_SIZE` (50, lib/pagination.ts); the folder page fetches the first page server-side, ProblemList.tsx's "Load more" button fetches the rest client-side and appends. "Needs Revision" was deliberately left out of this — it's grouped into topic subfolders client-side (RevisionTopicGroups) and staleness/SM-2 rules already keep it naturally bounded, so it gets one larger single fetch (200) instead of learning to page too, which would have meant teaching the topic-grouping view to re-fetch and re-group on "load more."

Why cache dashboard stats with explicit invalidation instead of just a TTL, unlike the Needs Revision cache (2026-07-13)?
problemCount and the Easy/Med/Hard breakdown are cached in Redis (lib/dashboardStats.ts) the same way Needs Revision is, but a pure TTL alone would directly contradict the sync routes' own "refresh your dashboard in a few seconds" message — a user syncing and immediately checking their dashboard would see minutes-old counts. So the three places that actually change a user's problem count or difficulty mix (LeetCode sync, Codeforces sync, manual problem add) explicitly call invalidateDashboardStats() right after their writes succeed, deleting the cache key outright rather than waiting for it to expire. The 10-minute TTL still exists underneath as a safety net in case an invalidation call is ever missed by a future change, not as the primary way this stays correct. Problem *moves* and folder deletes that cascade-delete problems don't currently invalidate this cache — a known small gap, not wired up yet.

Why did syncing produce two folders for the same topic — "Binary Search" and "binary search," "Dynamic Programming" and "dp" (2026-07-13)?
getPrimaryTag() (lib/tagNormalizer.ts) picked a folder name by checking `Array.includes()` against fixed Title-Case tier lists (ADVANCED/INTERMEDIATE), an exact case-sensitive string match. LeetCode's GraphQL API returns tags already in that Title-Case taxonomy ("Binary Search", "Dynamic Programming"), so LeetCode syncs matched fine. Codeforces's API returns its own tag vocabulary — lowercase, and sometimes a different word entirely ("dp" instead of "Dynamic Programming", "binary search" instead of "Binary Search"). Those never matched the tier lists, so they fell through to `return normalized[0]`, i.e. the raw tag verbatim, which then went straight into `folder.upsert` as a brand-new folder name distinct from the one LeetCode syncing had already created for the same topic.

The fix has two layers. First, lib/tagNormalizer.ts now runs every tag through a new normalizeTag() before tier-matching: a TAG_ALIASES map translates known Codeforces-style spellings ("dp", "binary search", "dfs", "bfs", "hashing", "arrays", "trees", etc.) to the same canonical Title-Case name LeetCode already uses, and anything not in the map falls back to naive title-casing rather than being left as whatever case it arrived in. This closes the gap for every tag already known about, going forward, for both platforms.

Second, as defense-in-depth, lib/folderUpsert.ts adds findOrCreateFolder(userId, name, type): a case-insensitive lookup (Prisma `mode: "insensitive"`) that finds an existing folder regardless of casing before creating a new one. This catches anything normalizeTag() doesn't recognize (a brand new tag neither alias list nor tier list has ever seen) and also protects app/api/folder/create/route.ts, where a user could otherwise type "Google Prep" and "google prep" as two separate CUSTOM folders by hand. All three call sites (leetcode sync, codeforces sync, folder/create) now go through this instead of `prisma.folder.upsert`, which only matched on the exact `(userId, name, type)` unique constraint and had no case-insensitivity of its own.

Neither layer touches folders that already exist from before this fix — normalizeTag() and findOrCreateFolder() only change what happens on the *next* sync or create. Existing duplicate pairs needed a one-time migration of their own: POST /api/folder/merge-duplicates groups a user's current folders by `(type, name.toLowerCase())`, and for any group with more than one folder, keeps the oldest (by createdAt) and moves every FolderProblem link from the newer duplicates into it via upsert before deleting the duplicate folder rows. It's exposed as a "Clean up duplicate folders" button in ProfileForm.tsx rather than run automatically, since merging is a real data mutation (deletes rows) and should be something the user triggers deliberately, once, rather than something that fires silently on a page load. It's idempotent — running it again when nothing's left to merge just returns zero counts.

## Testing (2026-07-13)

Why Vitest, and why only 4 files tested instead of the whole app?
No test runner existed in this project at all before this. Vitest was picked over Jest because it needs near-zero config for a TypeScript project (no ts-jest/babel setup), runs fast, and its `vi.mock` API is what mockRedis below relies on. Rather than trying to cover every route (most of which are thin glue: auth check → Prisma call → JSON response, and would mostly be testing Prisma/Next.js itself), the four files covered here were chosen because they're the actual *logic* in this codebase — the parts that are easy to get subtly wrong and hard to verify just by reading them or clicking through the UI once. Route handlers were left uncovered for now; that's a real gap, not an oversight — see the "still open" note at the end of this section.

lib/spacedRepetition.ts → lib/__tests__/spacedRepetition.test.ts
Covers computeSM2(), the SM-2 scheduling function. Tests: the first review schedules 1 day out, the second schedules 6 days out, the third-and-later reviews multiply the previous interval by the ease factor, a failed recall (quality < 3) resets both repetitionCount and the interval back to 1 day, Easy raises the ease factor while Again/Hard lower it, the ease factor never drops below SM-2's 1.3 floor even after 20 consecutive failures, and out-of-range quality inputs (99, -10) get clamped instead of producing nonsense. This function decides when a problem resurfaces in "Needs Revision" — a silent bug here (e.g. the ease-factor floor missing, or the reset-on-failure branch not firing) wouldn't throw an error, it would just quietly mis-schedule reviews forever, which is exactly the kind of bug that's invisible until you're staring at three months of bad revision timing.

lib/tagNormalizer.ts → lib/__tests__/tagNormalizer.test.ts
Covers normalizeTag() and getPrimaryTag() — the exact functions responsible for the duplicate-folder bug fixed earlier in this session. Tests: known Codeforces aliases ("dp", "graphs", "dfs and similar") map to LeetCode's canonical wording, mixed-case input normalizes identically regardless of source casing, an unrecognized tag still gets consistent title-casing instead of being left verbatim, tier priority (Advanced > Intermediate > fallback) is respected, and — the test that most directly guards against this bug ever coming back — a LeetCode-style tag list and a Codeforces-style tag list for the same topic resolve to the exact same primary tag. This is the one test file in this batch written specifically because of a bug that already happened once; it exists to make sure it can't happen again silently.

lib/rateLimit.ts → lib/__tests__/rateLimit.test.ts
Covers checkCooldown(), mocking the Redis client entirely with `vi.mock("@/lib/redis")` rather than hitting real Upstash — this test suite has no network access and shouldn't need any to verify pure logic. Tests: a fresh key allows the request, a key that already exists (simulating Redis's `SET NX` returning null) correctly rejects it, and the Redis key is built from both namespace and id together (so `sync:leetcode` cooldowns for two different users, or two different cooldown types for the same user, never collide). The atomicity guarantee itself (that `SET NX` closes the race window a GET-then-SET would leave open) isn't something a unit test can prove — that's a property of Redis's own command semantics, not this function's code — so this test suite verifies checkCooldown() calls Redis correctly, not that Redis itself is atomic.

lib/pagination.ts → lib/__tests__/pagination.test.ts
This one required a small refactor first: the "fetch pageSize+1 rows, slice off the extra one, compute hasMore/nextCursor" logic was previously duplicated almost identically in app/api/folder/[folderId]/problems/route.ts and app/folders/[folderId]/page.tsx. Pulled it into one exported paginate() function so there's a single place this logic can be wrong, and so it's testable without a database at all. Tests cover the edge case that actually matters here: a folder with exactly `pageSize` problems must report hasMore=false (there's no pageSize+1'th row for Prisma to have returned), which is easy to get off-by-one wrong and would otherwise show a "Load more" button that loads nothing. Also covers the empty-folder case, a partial page, and confirming nextCursor always points at the last row actually shown to the user — not the lookahead row used only to detect a next page exists.

What's still not tested (an honest gap, not hidden)
No test yet covers Prisma-touching code directly (any API route, revisionEngine.ts, dashboardStats.ts, goalEngine.ts) — those would need either a real test database or a mocked Prisma client, which is a bigger setup investment than this round covered.

Update (2026-07-19): the ownership/IDOR gap flagged above now has a first pass of coverage — see "Route-level input validation + IDOR regression tests" below. Still open: revisionEngine.ts, dashboardStats.ts, goalEngine.ts, and the remaining routes not yet in ownership.test.ts (folder/delete, folder/nest, folder/reorder, problem/move, problem/dismiss-revision, problem/view, goal/delete, goal/recurring/*) all still rely on manual code review rather than an automated regression test, even though they follow the same `{ id, userId }` pattern already verified elsewhere.

Route-level input validation + IDOR regression tests (2026-07-19)
Two related gaps closed together, since fixing one made the other easier to test. First: every mutating route used to hand-roll its own body checks (`if (!name?.trim())`, `if (!VALID_TIERS.includes(tier))`, etc.) — each slightly different in thoroughness, none catching a wrong *type* (a number where a string was expected) rather than just a missing field. Replaced with `lib/validation.ts`: one `parseBody(schema, body)` helper plus one zod schema per route, wired into every route in app/api that accepts a JSON body (auth/register was left alone — its existing manual checks are already thorough and it's the one route that shouldn't share validation code with the PrismaAdapter-free credentials flow). Behavior is unchanged for well-formed requests; malformed ones now get a consistent 400 with a real reason instead of whatever each route happened to check for. `problem/add`'s platform/difficulty enum checks moved into the schema itself instead of a manual `VALID_PLATFORMS.includes(...)` array, which removed the last hand-rolled enum check in the codebase.

Second: added app/api/__tests__/ownership.test.ts, the coverage DECISIONS.md's "What's still not tested" section explicitly called out as the highest-value gap. Mocks `@/auth` and `@/lib/prisma`, then asserts two things per route: the Prisma query actually sent is scoped to the caller's own userId (not just the resource id), and when that scoped query finds nothing — i.e. the resource belongs to someone else — the route 404s instead of performing the write. Covers folder/rename, folder/set-tier, notes/save, goal/toggle, and revision/toggle as a representative sample of the `findFirst`-then-`update` and `updateMany`-with-userId patterns used throughout; not every route yet (see the "still open" note above). This tests the query shape sent to Prisma, not a real database — it would not catch a case where the Prisma client itself misapplies a `where` clause, only a case where the *route code* stops asking for the userId scoping in the first place, which is the actual regression this guards against (a future edit that "simplifies" a query back down to `findFirst({ where: { id } })` and reintroduces the exact IDOR bug the original `{ id, userId }` pattern existed to prevent).

Couldn't run either test file in this environment — vitest's own dependency chain (tinyglobby/fdir) fails to resolve here, unrelated to anything in this change. Verified by hand instead: re-read every edited route file end to end and traced each test's mocked call shape against the route's actual Prisma call. Run `npm test` locally to confirm before trusting this further.

Why group the dashboard's folders into labeled sections instead of one flat grid (2026-07-13)?
getPrimaryTag() can realistically produce 30-40+ distinct AUTO folder names across LeetCode's and Codeforces' combined tag vocabularies, and FolderGrid.tsx was rendering every one of them — AUTO, CUSTOM, pinned — as an identical tile in one undifferentiated grid. At that count it reads as visual noise with no hierarchy, not a dashboard. Fixed by reusing the same tier classification getPrimaryTag() already applies when *assigning* a folder (lib/tagNormalizer.ts's ADVANCED/INTERMEDIATE lists, now exported plus a new getFolderTier() helper) to *group* folders for display: a pinned row (Needs Revision, Revision) on top, then Advanced / Intermediate / Fundamentals & Other / Custom as labeled sections underneath. Reusing the existing tier lists rather than inventing a separate grouping scheme means the section a folder lands in can never disagree with why it was created in the first place. getFolderTier() compares case-insensitively on purpose — a folder created before the tag-normalization fix (or not yet run through merge-duplicates) still lands in the correct section instead of being miscategorized into "Fundamentals" just because its stored name happens to be lowercase.

Why keep drag-to-reorder scoped to within a section rather than dropping it or leaving it dashboard-wide (2026-07-13)?
A global free-form drag order and tier-based grouping don't really coexist — dragging "Graph" out of Intermediate and on top of a Custom folder wouldn't mean anything once section membership is derived from the folder's own name/type, not from where it visually sits. Rather than removing drag entirely (losing the ability to prioritize, say, "Dynamic Programming" above "Recursion" within Advanced), FolderGrid.tsx's reorderLocally() now checks sectionOf(dragged) === sectionOf(target) before allowing a reorder — dragging within a section works exactly as before, dragging across sections is a no-op. The underlying persisted order (`order` column, /api/folder/reorder) is still one flat list; only which drags the UI allows changed, not the persistence mechanism itself.

Why add Folder.tierOverride instead of leaving section membership purely automatic (2026-07-13)?
The very next thing asked after grouping folders by tier was "can't we drag Intermediate into Advanced, topic-wise?" — a fair point: getFolderTier()'s classification is LeetCode's own official difficulty framing, not necessarily how a specific user personally experiences that topic. Rather than pick one or the other, folders now support an optional manual override: dragging a folder from one tier section into another (still blocked for Custom, both directions — Custom isn't organized by perceived difficulty at all) sets tierOverride on that folder via POST /api/folder/set-tier, and sectionOf() in FolderGrid.tsx checks tierOverride first, falling back to the automatic getFolderTier(name) classification only when no override is set. This is deliberately per-folder, not a change to getPrimaryTag()'s tier lists themselves — it doesn't affect which AUTO folder a newly-synced problem gets filed into, only which dashboard section an existing folder visually sits in for this user. The API route scopes the update to `type: "AUTO"` in the same `updateMany({ where: { id, userId, ... } })` ownership-scoping shape used everywhere else in this app, so a Custom folder can't be re-tiered even by a crafted request, and one user can't touch another's folder.

Why compare against the `folders` prop instead of `items` state to decide whether to call the reclassify API (2026-07-13)?
`items` already reflects the optimistic local update made mid-drag (the tile visually jumped sections before the drag even ended), so by the time persistOrder() runs, items and the "did this actually change" question would always agree with each other trivially. Comparing the dragged folder's *current* computed section (from `items`, post-drag) against its *original* section (recomputed from the untouched `folders` prop as it was before this drag started) is what actually answers "did the tier change during this drag" — and skips the API call entirely on a same-section reorder, which is the common case.

Why per-resource cooldowns (folderId/problemId) instead of per-user like the sync routes, for the remaining mutation routes (2026-07-13)?
folder/rename, folder/delete, problem/move, and notes/save now all call checkCooldown() too, but scoped to the specific folder or problem being acted on rather than the user as a whole (2-5 second TTLs). This is a different kind of protection than the sync routes' 5-minute per-user cooldown: sync protects an external API (LeetCode/Codeforces) and an expensive multi-step DB pipeline from repeated full runs, so blocking the *user* for a while is correct. These four routes are single, cheap DB writes with no external API involved — the only failure mode worth guarding against is a double-click or a client retry firing the exact same mutation twice in the same second. A per-user cooldown here would be wrong: it would block a user from renaming a *second, different* folder just because they renamed one a moment ago, which isn't a real problem, just an artificial throttle. Scoping the Redis key to the specific resource id means only "do the same thing to the same thing again immediately" gets rejected.

Why did problem/move's cooldown use MOVE_COOLDOWN_S = 2s specifically, and why is it keyed on problemId rather than (fromFolderId, toFolderId)?
Keying on problemId (not the folder pair) means moving the *same* problem twice rapidly is blocked, but moving five different problems into the same folder back-to-back — completely normal cleanup behavior — is never affected, since each has its own problemId key. 2 seconds (vs. 3-5s for rename/delete) reflects that this is purely a double-drag guard on a fast, frequent UI action, not a meaningfully destructive one — no data is lost if it fires twice, a link just gets created/removed twice in a row.

Why does dismissing a problem from Needs Revision/Revision not run SM-2 (2026-07-14)?
Added a checkbox (and made clicking the problem title do the same thing) inside these two folders specifically — POST /api/problem/dismiss-revision — that removes a problem from whichever of the two folders it's in. Deliberately does not touch easeFactor/intervalDays/revisionCount: this is for a user who doesn't want to go through the Again/Hard/Good/Easy rating, just wants to say "I've seen this, stop showing it to me" without committing to a graded self-assessment. For Needs Revision, that means clearing nextReviewAt (so getStaleProblems() falls back to the flat 30-day heuristic, restarted from lastViewedAt = now) rather than running real spaced-repetition math; for the starred Revision folder, it's just removing the FolderProblem link, identical in effect to clicking the ★ toggle again. Both cases also delete the FolderProblem row directly rather than waiting for Needs Revision's hourly cache to catch up (see the earlier Redis caching entry) — dismissal is meant to feel immediate, not lag up to an hour.

Why did the frontend components (RenameFolderButton, DeleteFolderButton, ProblemList's saveNote/moveProblem) need changes for this, when the API routes already return a 429 on their own?
Before this, all four call sites did `await fetch(...)` and used the response for nothing — no `res.ok` check, so a 429 (or any other error) was silently swallowed: the modal would close, the UI would optimistically act like it worked, and the actual mutation just... didn't happen, with zero indication to the user why. Added a `res.ok` check to each that shows `toast.error(data.error)` and returns early on failure — the same pattern ProfileForm.tsx's sync handlers already used. Without this, adding server-side cooldowns purely on the backend would have added confusing, invisible failures instead of protecting anything.

Why add Sentry (@sentry/nextjs) now (2026-07-13)?
Before this, error visibility was inconsistent and manual: the LeetCode sync route had a try/catch that logged to console.error and an ActivityLog row; the Codeforces sync route (a real gap, found while doing this) had no try/catch around its background sync at all — an unhandled throw there was a silent unhandled rejection inside next/server's after(), no ActivityLog row, no record anywhere. Everywhere else, a broken Server Component render or an uncaught exception in an API route just became generic Next.js error output with nothing captured. Sentry.captureRequestError wired into instrumentation.ts's onRequestError hook catches Server Component/Server Action errors Next's own error boundary would otherwise swallow; Sentry.captureException was added explicitly to both sync routes' catch blocks (writing the Codeforces one for the first time, matching LeetCode's existing shape) so a failed background sync surfaces somewhere a person would actually see it, instead of only existing as a console line that scrolls away or a DB row nothing watches.

Why separate instrumentation.ts / instrumentation-client.ts / sentry.server.config.ts / sentry.edge.config.ts instead of one Sentry.init() call?
This follows Sentry's own Next.js SDK convention rather than inventing a different structure: instrumentation.ts's register() hook runs once at server startup and dynamically imports the server or edge config depending on which runtime is executing (NEXT_RUNTIME env var), since the two runtimes need different underlying integrations; instrumentation-client.ts is the modern (Next.js 15.3+) equivalent for the browser bundle, catching errors that never touch the server at all (a broken Client Component render, an unhandled rejected fetch) — a failure class server-side tracking alone would completely miss. sentry.edge.config.ts exists even though this app has no custom middleware.ts today, purely because instrumentation.ts's edge branch expects something to import; cheap to keep in place versus needing it the moment a real middleware.ts is added later.

Why is this safe to ship before a real Sentry account/DSN exists?
Sentry.init() with an empty/undefined `dsn` no-ops silently — it doesn't throw, it just doesn't send anything. Same for next.config.ts's withSentryConfig: without SENTRY_AUTH_TOKEN/SENTRY_ORG/SENTRY_PROJECT set, it skips the source-map-upload step rather than failing the build. All five new env vars were added to .env.example as blank-by-default, and CI's placeholder-env-var job (see the earlier CI entry) needed no changes because of this — the build succeeds identically whether or not Sentry is actually configured. Actually seeing errors in a Sentry dashboard requires creating a free Sentry.io project and setting SENTRY_DSN / NEXT_PUBLIC_SENTRY_DSN locally and in your deployment platform's env vars — a one-time step outside what code alone can do.

Why add `npm test` to the CI workflow now, when the test suite itself was built in an earlier round?
It was a genuine gap: the Vitest suite (lib/__tests__/) was built and verified passing locally, but nothing was wired into .github/workflows/ci.yml to actually run it on every push/PR — meaning a future change could silently break spacedRepetition.ts or tagNormalizer.ts and CI would stay green regardless, defeating half the point of having tests at all. Added `npm test` (→ `vitest run`) as its own step, between lint and build, so it fails the workflow the same way a lint or build error would.

Why cap folder nesting at exactly one level, and why is it enforced with two checks instead of a real depth counter (2026-07-15)?
Folder gained a self-relation (parentId → parent/children) so folders can be dragged inside each other from the dashboard. Unlimited depth was considered and deliberately rejected: a true tree needs recursive rendering (FolderGrid.tsx, the folder detail page, breadcrumbs) and edge cases like "can't drop a folder into its own descendant," none of which this app's actual use case needs — a personal LeetCode/Codeforces tracker organizing topics doesn't need three levels of subfolders. One level (a folder can hold children, but a child can't itself have children) covers the real want ("put this folder inside that one") with a much simpler shape. It's enforced in POST /api/folder/nest with two checks rather than walking up a chain: (1) the folder being nested can't already have children of its own, (2) the target parent can't itself already have a parentId. Together these also rule out cycles for free — the moment A becomes parent of B, check 1 permanently blocks A from being nested under B, and check 2 permanently blocks B from being nested under A, so a 2-folder cycle can never form without a depth counter or recursive ancestor walk.

Why any folder type (AUTO or CUSTOM) can be nested, but pinned folders can't be dragged at all (2026-07-15)?
Nesting was scoped to "any folder" rather than "custom folders only" — a topic AUTO folder like "Binary Search" nested under "Arrays" is a legitimate way to express "this is a subtopic of that," independent of the tier system (Advanced/Intermediate/Fundamentals), which keeps working unchanged for whichever folders stay top-level. Nesting and tiering are orthogonal: a nested child keeps its own type/tierOverride, it just doesn't get its own top-level tile or section slot while nested (FolderGrid.tsx renders it as a small chip inside its parent's tile instead). "Needs Revision" and "Revision" were never a concern here since they're already excluded from `draggable` entirely (see the earlier "why can't these be dragged" entry) — the same exclusion means they can never be dragged to nest, nor can anything be dropped onto them as a nest target, with no extra code needed.

Why did the AI hint profile's tag stats need the same normalization fix as folder names (2026-07-16)?
lib/profileBuilder.ts's buildUserProfile() grouped by the raw tag string on each Problem row, and Problem.tags stores whatever a platform sent verbatim — normalizeTag() was only ever applied at folder-assignment time (getPrimaryTag(), during sync), never to this aggregation. So a user solving on both platforms had LeetCode's "Dynamic Programming" and Codeforces's "dp" silently counting as two separate topics in their strengths/weaknesses — the exact class of bug normalizeTag() already existed to fix, just not applied here. Fixed by normalizing each tag before it's added to tagStats (deduping per-problem so a tag appearing twice on one problem's tags array doesn't double-count).

Second migration drift gap found via a full reset (2026-07-16): lcUsername, cfHandle, and emailVerified existed in schema.prisma and had been in active use, but no migration file ever actually created them on the User table — they'd been added directly to a live database at some point (almost certainly `prisma db push`, which updates a database without recording a migration). Invisible as long as that same database kept getting reused; surfaced immediately by a full `prisma migrate reset`, which rebuilds strictly from migration files and has no way to know about a change that was never written to one. Fixed with a new migration (20260716130000_add_missing_user_columns) that actually adds the three columns. Same root cause and same lesson as the entry below, just a second, independent instance of it — worth treating as a pattern rather than a one-off: any schema change needs to go through a migration file to actually be part of this project's history, `db push` during quick iteration has to be followed up with `prisma migrate dev` before moving on, or it silently disappears from anyone/anything that rebuilds from scratch.

Correction to the 2026-07-12 "Migration drift recovery" entry (2026-07-16): that entry's fix removed the CREATE TYPE statements from 20260712120000_platform_difficulty_enums/migration.sql because, on the database being debugged at the time, the enum types already existed from an earlier partial run. That was a patch for one database's specific state, not a real fix — running the migration history from scratch on any other database (a fresh clone, a new environment) hit the same file and failed immediately, since nothing ever created the types there. Rewrote it to wrap both CREATE TYPE statements in a DO $$ ... EXCEPTION WHEN duplicate_object THEN null $$ block, so the same file is correct whether or not the types already exist, instead of assuming one or the other. Lesson underneath the original lesson: "fix the migration so it runs" and "fix the migration so it runs *from a clean database*" are different bars, and only the second one is actually safe to keep in the repo's history for anyone (including a future fresh setup) to run.

Why a separate RecurringGoal model instead of a boolean flag on DailyGoal (2026-07-16)?
Added "Weekly Goals" — a template list (components/WeeklyGoalsList.tsx) that spawns a fresh, unchecked DailyGoal instance every day automatically, so a goal doesn't have to be re-typed each morning. A template ("this repeats indefinitely") and a specific day's completion record ("was this checked off last Tuesday") are different concepts, and cramming both into one row would make "stop this from repeating" ambiguous with "delete today's checkbox" — exactly the kind of question the design discussion with the user resolved explicitly (removing a recurring goal only stops future days from getting a new instance; existing days keep their own state, since each DailyGoal stores its own title/done rather than looking it up through the relation). DailyGoal.recurringGoalId links a spawned instance back to its template and is null for one-off, single-day goals exactly as before this feature existed.

Why does ensureTodayRecurringGoals() run on every goals-page load instead of a scheduled job (2026-07-16)?
Same reasoning as syncNeedsRevisionFolder on the dashboard: this app has no job-scheduling infrastructure, so instead of a cron spawning each day's instances at midnight, the goals page just creates any of today's rows that don't exist yet for the user's active recurring goals, every time it loads. createMany + skipDuplicates leans on the (recurringGoalId, targetDate) unique constraint, so this is safe to call more than once in the same day (two tabs, a refresh) — it's a no-op once today's rows already exist.

Why relay Codeforces's "rank" string directly instead of computing a tier label from rating ourselves (2026-07-16)?
Added getCodeforcesRating() (CF's user.info) and getLeetCodeContestRating() (LC's userContestRanking GraphQL query, separate from submitStats) alongside the totals fetch, same Redis-cached-external-call shape. Codeforces already returns a rank string ("specialist", "expert", etc.) computed from rating — re-implementing those tier thresholds locally would risk quietly disagreeing with Codeforces's own labeling if they ever adjust them, for no real benefit over just relaying what they already say (capitalized for display). LeetCode's contest rating comes back null, not an error, for anyone who's never entered a rated contest — the common case for someone who only uses the practice problem set — so it's rendered as "No rated contests yet" rather than treated as a fetch failure.

Why a new lib/unifiedProfile.ts instead of extending profileBuilder.ts further (2026-07-16)?
profileBuilder.ts's job is building the AI hint prompt's personalization context (top-5/bottom-3 tags, recent problems) — a different consumer than a full per-topic, per-platform, per-difficulty breakdown for a dashboard view. getUnifiedTagBreakdown() applies the same normalize-before-grouping fix, but keeps every topic (not just the extremes) and splits each one by platform, which the hint prompt has no use for. getLeetCodeActualTotals() fetches real platform-wide Easy/Medium/Hard counts via LeetCode's own submitStats GraphQL field — independent of the sync's own 50-submission cap — so the dashboard can honestly show "synced vs actual" instead of quietly presenting a partial number as the whole picture. Cached in Redis for 30 minutes, same cache-aside shape as dashboardStats.ts, since it's an external call that doesn't need to hit LeetCode on every profile page view.

Why does dropping onto a folder's center nest, but the top/bottom edges still reorder — and why is un-nesting a button, not a reverse drag?
FolderGrid.tsx's existing drag model just moved the dragged tile to wherever it was hovering, with no distinction between "you're aiming at this position" and "you're aiming at this folder." Nesting needed a second, more deliberate gesture layered on top without breaking the first: `onDragOver` now reads the pointer's vertical position within the hovered tile's bounding box — the middle band (30%-70% of the tile's height) sets it as a nest target (visually highlighted with a ring), the top/bottom edges fall through to the original reorder/retier logic unchanged. Un-nesting (moving a child back to top-level) is a plain × button on its chip instead of a reverse drag gesture — dragging a small chip back out to a specific dashboard section would need its own drop-zone handling for a case simple enough not to need one; a button is fully discoverable and immediate.

Why does deleting a folder that has children need no special-case code, and why does moving a child's problems into its parent (or vice versa) already just work (2026-07-15)?
Both were explicit requirements, not incidental. Deleting a parent: Folder.parentId is declared with `onDelete: SetNull`, so Postgres itself promotes any children back to top-level (parentId → null) as part of the same DELETE that removes the parent row — the existing move_problems/delete_problems logic in app/api/folder/delete/route.ts only ever touches the deleted folder's *own* direct FolderProblem links, so a child folder's problems are never touched by its parent being deleted, and the child survives as an ordinary top-level folder afterward, independently deletable/renameable exactly like before it was ever nested. Moving problems between a parent and its child: DeleteFolderButton's "move problems to another folder" dropdown is populated from `allFolders`, a flat, nesting-unaware list of every folder the user owns (queried without a parentId filter) — a parent is just one more entry in a child's dropdown and vice versa, so transferring a child's problems into its parent (or a parent's problems into one of its children) before deleting either one already worked with zero nesting-specific code, since the move logic was never folder-hierarchy-aware to begin with.

Third migration drift gap found via the same full reset (2026-07-16): Problem's composite unique index on (userId, url) also existed on the pre-reset database but had no migration file creating it — same root cause as the enum-type and User-column gaps above (almost certainly `db push` at some earlier point), found the same way (a full `prisma migrate reset` rebuilding strictly from migration history exposed it immediately). `prisma migrate dev` auto-generated the fix this time rather than needing a hand-written migration, since Prisma could diff the current schema.prisma against the (now correctly-rebuilt) migration history and see the constraint was genuinely missing — applied as 20260716134115_fix_problem_userid_url_unique_constraint. Three independent instances of the identical pattern in one debugging session is the strongest evidence yet that this project needs a deliberate audit (diff schema.prisma's declared shape against a fresh `migrate reset`'s actual result, once, end to end) rather than continuing to find these one at a time whenever something else forces a reset.

Why a getGoalStreak() count instead of just reading the weekly graph's percentages (2026-07-16)?
The weekly graph only ever shows the current Monday-Sunday window, so it can't answer "how many days in a row" once a streak crosses a week boundary. getGoalStreak() (lib/goalEngine.ts) walks backward day-by-day from today, using the same startOfDay/addDays helpers already in the file, counting a day as part of the streak only if it had at least one goal and all of them were done — a day with zero goals breaks the streak the same as a day with unfinished ones, since the point is "showed up and finished what I set," not just "nothing failed because nothing was tried." Today is a special case: if it isn't fully done yet (still in progress, or no goals added yet today), it's skipped rather than treated as a break, so an unfinished today doesn't zero out a real streak built through yesterday — once today is itself complete, it counts like any other day. Capped at a 400-day lookback purely so the query and the backward walk both have a bound, not because a real streak is expected to reach that.

Why add real email/password login (Credentials provider) alongside Google instead of Google-only (2026-07-16)?
Until now sign-in was Google-only — fine for the one account this was built and tested against, but a real blocker for anyone without (or unwilling to use) a Google account. Added a `password` column on User (nullable — Google-only accounts simply never have one set) and a Credentials provider in auth.ts alongside the existing Google provider. Account *creation* for credentials happens only in /api/auth/register (hashes with bcrypt, 10 rounds, writes the User row directly); the Credentials provider's authorize() only ever reads and compares, it never creates a user — keeping it fully decoupled from the PrismaAdapter, which remains solely responsible for Google's account linking. A login attempt with an email that either doesn't exist or belongs to a Google-only account (password is null) is rejected identically rather than distinguished, so failed logins can't be used to enumerate which emails are registered. This works under the existing `session: { strategy: "jwt" }` setup already in place — Credentials providers aren't compatible with database sessions in NextAuth v5, only JWT, so no session-strategy change was needed, just an addition to it.

Why split Settings into its own page instead of folding it further into Profile (2026-07-16)?
Profile had been accumulating two different kinds of content: identity + progress (avatar, name, UnifiedProgressView's stats/tags/ratings) and account configuration (LC/CF handles, buffer day, duplicate-folder cleanup) via ProfileForm. Those serve different intents — one is "how am I doing," the other is "change how this app behaves for me" — so ProfileForm moved wholesale to a new /settings page, restyled into labeled sections (Sync Sources, Preferences, Maintenance, Account) with consistent card borders instead of one long undifferentiated stack of identical-looking boxes. /profile now links to /settings and vice versa isn't needed since Settings doesn't need to link back. Settings is also where Sign Out finally lives — there had been no sign-out control anywhere in the UI until now, only the implicit `/api/auth/signout` NextAuth route nobody had a link to; it's a plain server action form (`signOut({ redirectTo: "/login" })`) next to the account's name/email, not tucked away, since ending a session is common enough to deserve a visible, obvious control.

Why does the login page hide the Navbar, and why did app/page.tsx get replaced with a redirect (2026-07-16)?
Adding a real Login/Register form to /login made it clear the page was still getting a full Dashboard/Goals/Profile/Settings nav bar rendered above it from the root layout — links to authenticated pages sitting above a logged-out user's login card, each of which would just bounce them right back to /login anyway. Navbar.tsx became a client component that reads usePathname() and renders nothing on "/login" rather than adding per-page layout logic elsewhere. Separately, app/page.tsx (the site root, "/") turned out to be a stale, forgotten duplicate of a much older version of the Profile page — it rendered ProfileForm without passing the now-required `bufferDay` prop, meaning it had likely already been a type error for a while without anyone noticing, since nothing links to "/" directly in normal use. Replaced it with a plain redirect (to /dashboard if signed in, /login otherwise) — "/" doesn't need its own view, it only needs to send people to the page that actually matters.

Codebase-wide tidy-up pass, no behavior changes (2026-07-16)
Went through every file in lib/, components/, and app/ looking for code that was doing the same thing with more lines or more repetition than it needed, without touching what anything actually does. The one real structural win: lib/unifiedProfile.ts's three external-API fetchers (getLeetCodeActualTotals, getLeetCodeContestRating, getCodeforcesRating) were each hand-rolling an identical "check Redis, fetch, cache on success, swallow errors as null" shape — pulled into one shared withCache() wrapper, so that logic exists in exactly one place instead of three near-identical copies that could quietly drift apart from each other over time. Smaller mechanical cleanups: app/api/problem/add/route.ts had several early-return error responses needlessly spread across 4-5 lines each where every other route in this app writes them on one line — collapsed for consistency (137 lines → 85, same status codes and messages throughout); the same one-line collapse applied to a stray multi-line check in sync/leetcode/route.ts; app/folders/[folderId]/page.tsx had a couple of Prisma queries and one JSX tag written across unnecessarily many lines for how little they actually do — collapsed the same way; components/ProblemList.tsx's move-to-folder dropdown was filtering allFolders by the same two conditions twice (once to render the list, once just to check if it was empty) — computed once into a local variable and reused. Most of the codebase (goalEngine.ts, revisionEngine.ts, dashboardStats.ts, tagNormalizer.ts, FolderGrid.tsx, ProblemForm.tsx, and the rest of the API routes) was already about as tight as it can get without cutting the comments explaining *why* something is written the way it is — those comments were left alone on purpose, since "fewer lines" isn't the same goal as "easier for the next person (or an interviewer) to actually understand."

Three scale/correctness gaps fixed (2026-07-17): LeetCode's recentAcSubmissionList was called with a hardcoded `limit: 50` — an arbitrary number this app's own code chose, not a documented LeetCode ceiling — so anyone with a bigger solved history than that only ever got their most recent 50 synced. Raised to 10000 (same "ask for far more than anyone needs" pattern the Codeforces route already used with `count=10000`); LeetCode's own server still decides the real cap, so this can only ever match or beat the old behavior, never regress it. Couldn't verify the real outcome from this environment (no outbound network access in the sandbox this was built in) — confirm on the next real sync.

Codeforces sync wrote every problem to the database one at a time in a plain for-loop, unlike LeetCode's sync which already had bounded-concurrency tag fetching. Pulled the `mapWithConcurrency` helper out of sync/leetcode/route.ts into lib/concurrency.ts so both routes share one implementation, then used it for Codeforces's DB writes (8 in flight at once, same number LeetCode already used for tag fetches). This isn't safe to do naively, though: findOrCreateFolder's findFirst-then-create isn't atomic, so two concurrent writes racing to create the same not-yet-existing AUTO folder could create it twice. Fixed by splitting the work into two phases — first, sequentially ensure every *distinct* topic folder this batch needs already exists (a handful of folders, not one per problem, so staying sequential here is cheap), then run the actual per-problem upserts concurrently once every folder they could possibly need is guaranteed to already exist.

Needs Revision's "topic subfolder" view (RevisionTopicGroups.tsx) only ever received a single hard-capped batch of 200 stale problems with no way to see anything past it. Wired it the same way ProblemList.tsx's own "Load more" already worked: it now accepts folderId/initialHasMore/initialNextCursor, keeps its own loadedProblems state, and a "Load more problems" button (shown below the topic tiles, not inside an open topic) fetches the next page from the same /api/folder/[folderId]/problems route every other folder already uses — no backend changes needed, since that route was already folder-agnostic and simply had never been called for this particular view. The 200-per-page size itself was left unchanged, since it already covers the realistic common case in one fetch (the note on this from 2026-07-13 that staleness/SM-2 rules keep this naturally bounded still holds); the fix is that a user who does exceed it now has a way to see the rest instead of hitting an invisible wall.

AI hint retrieval made branch-aware instead of flat (stage 1 of 3, 2026-07-19): findSimilarSolvedProblems() in app/api/hint/rag/route.ts used to keyword/tag-score a user's *entire* solved history against the pasted problem every single time — no awareness of topic at all, so a DP problem could surface an unrelated String match just because a couple of title words happened to overlap. Rather than inventing a new classification scheme, added classifyTopicFromText() to lib/tagNormalizer.ts that reuses the exact TAG_ALIASES table already used to normalize LeetCode/Codeforces tags at sync time — it scans the pasted problem statement (longest phrase first) for any known alias, plus a small extra table of ~7 phrasings that imply a topic without naming it ("subsequence" → Dynamic Programming, "shortest path" → Graph, etc.). This had to work off raw text rather than a supplied tags array because AIAssistant.tsx, the only real caller of this route, always sends `tags: []` — there was never a tags array to filter on in actual usage, despite the route accepting one as a parameter. findSimilarSolvedProblems() now scores against a scoped pool: the user's problems in that exact classified topic first; if fewer than 2 exist, widens to every problem in the same tier (Advanced/Intermediate/Fundamentals, via the existing getFolderTier()); if still fewer than 2, falls back to the original full-history scan. A wrong guess or a topic the user hasn't practiced yet still returns the old behavior, never an empty result — this can only narrow/improve the candidate pool, not regress it. Deliberately stopped here for stage 1: profileBuilder.ts's global strengths/weaknesses and pgvector-based semantic search were both left untouched, on the same "ship the deterministic free version first, add real infra only if it proves insufficient" pattern the rest of this app already follows.

Why two `npm audit` findings were left unfixed instead of running `--force` (2026-07-19)?
`npm audit fix` (no `--force`) closed 5 of 7 moderate advisories cleanly. The remaining 2 both only have a `--force` fix path, and both are in dependency chains this app never actually exercises at runtime. `@hono/node-server`'s advisory (a static-file-serving middleware bypass via repeated slashes) is pulled in transitively by `@prisma/dev` — Prisma's local dev-database tooling, not the `@prisma/client`/`PrismaPg` runtime path this app's routes actually use; DevSync never serves static files through Hono. `postcss`'s advisory (XSS via unescaped `</style>` when stringifying attacker-controlled CSS) is nested inside Next.js's own vendored copy (`node_modules/next/node_modules/postcss`), not a build script this app calls directly — Tailwind compiles from static class names at build time, never from runtime user input, so there's no attacker-controlled CSS ever reaching a stringifier here either way. Running `--force` would have downgraded `prisma` 7 → 6.19.3 (losing the driver-adapter model `lib/prisma.ts` is built around — see the earlier "Why PrismaPg adapter" entry) and `next` 16 → 9.3.3 (seven major versions back, losing the App Router entirely) to patch two advisories in code paths that never execute. Judged not worth it: left unfixed, documented here instead of silently ignored, and worth re-checking with `npm audit` again next time either package gets a routine version bump for unrelated reasons.
